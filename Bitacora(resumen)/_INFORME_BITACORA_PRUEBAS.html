f. CONCLUSIONES

  1. Sistema funcional: Todas las funcionalidades principalesdel sistema operan correctamente según los requerimientos funcionales.
  2. Validaciones работают: Las validaciones de datos (email, fecha, hora, conflicto) funcionan adecuadamente.
  3. Flujos completos: Los flujos de Registro → Login → Cita → Confirmación se ejecutan sin errores.
  4. Rendimiento aceptable: El sistema soporta hasta 50 usuarios simultáneos sin degradación crítica.
  5. Límites identificados: El sistema comienza a mostrar degradación por encima de 50 usuarios concurrentes.
  6. Compatibilidad de roles: Los tres roles (Paciente, Médico, Administrador) funcionam correctamente.

g. RECOMENDACIONES

  1. Producción: Para producción, se recomienda escalar el servidor o implementar balanceo de carga si se esperan más de 50 usuarios concurrentes.
  2. Debug: Desactivar modo debug en producción para mejorar tiempos de respuesta (de ~2s a < 500ms).
  3. Seguridad: Implementar autenticación JWT completa para todos los endpoints protegidos.
  4. Monitoreo: Agregar sistema de monitoreo en tiempo real para CPU, RAM y métricas de rendimiento.
  5. Documentación: Mantener esta bitácora actualizada para futuras pruebas.

h. LECCIONES APRENDIDAS

  1. Compatibilidad de versiones: Los archivos JMX pueden tener problemas de compatibilidad entre versiones diferentes de JMeter. Siempre verificar la versión antes de crear o abrir archivos.
  2. Configuración de headers: Es importante incluir HTTP Header Manager con Content-Type: application/json para solicitudes POST con JSON body.
  3. Autenticación: Para pruebas completas, se requiere manejar la autenticación JWT (tokens) correctamente.
  4. Monitoreo externo: JMeter no muestra uso de CPU/RAM del servidor; se necesita Task Manager o PerfMon.
  5. Documentación: Mantener evidencia de todas las pruebas ejecutadas es fundamental para el entregable final.
